## My scope

Standout feature for the Enerparc Digital Twin: a set of O&M agents that act on Max's model output.
Build my own isolated folders; do not touch Max's `backend/solar_twin/` or Tanmay's main frontend.

- Backend: `backend/agents/` (mine).
- Frontend: `test 2/project/` (Claude Design UI, served directly by FastAPI as static files). `frontend/agent-console/` is an older standalone Vite app (kept but not the primary UI).
- Outputs: `outputs/agents/` (findings.json, timelines/, emails/).

## The four pieces

1. Insight Engine / Root-Cause analyzer (always runs after the model): per inverter/issue -> severity, classification, root-cause, recommended action, priority rank, lost kWh, EUR (assumed tariff), qualitative repair-vs-replace. -> `outputs/agents/findings.json`.
2. Forensic Timeline ("inverter doctor"): reconstructs an inverter's life story from factor history + events.
3. Stakeholder Dispatch: Enerparc-tailored email per finding, routed by `contacts.py`; mock-send by default (writes to `outputs/agents/emails/`); optional real SMTP via `ENABLE_REAL_EMAIL=true`.
4. Plant Analyst Chatbot: NL Q&A grounded in the model payloads (Gemini tool-calling).

## Data contract (what I consume)

Primary source = Max's exported JSON payloads in `frontend/public/data/` (Plant A) and `frontend/public/data_plant_b/` (Plant B). Generated by `scripts/export_frontend_payloads.py` from `outputs/current_model_run/`. These are clean and guardrailed:

- `agent_context.json` - plant_headline, model_summary, guardrails, baseline_excluded_inverters, top_findings (evidence + recommended_next_action), top_events, answerable_questions.
- `plant_summary.json` - totals + yearly with reconciliation: `actual + lost + curtailment - overperformance = expected`. `lost_kwh` = performance loss EXCLUDING curtailment.
- `inverter_rankings.json` - enriched per inverter: rank, total_lost_kwh, lost_kwh_per_kwp, latest_factor, latest_relative_factor, worst_residual_z, samples, baseline_excluded, primary_status (critical/warning/watch/normal), status_color, primary_reason.
- `inverter_map.json` - spatial tiles (row, column, status, color) by physical adjacency.
- `events.json` - per-event records (large, ~9 MB): event_type (outage/acute_fault/fast_degradation/slow_degradation/curtailment/normal), severity, lost_kwh, residual z, error overlap.
- `inverters/<SAFE_ID>.json` - per inverter: summary, daily, factor_history, events.
- `model_run_metadata.json` - selected_model, train/calibration/score years, severity_thresholds, baseline_excluded_inverters.

Adapter falls back to `outputs/current_model_run/*.csv` if payloads are missing. All reads are cached in-process per plant via `@lru_cache`.

## Guardrails I MUST respect (from Max's agent_context)

- Every numeric claim is copied from exported values; never invent numbers.
- Unavailable values -> say "unknown".
- Baseline-excluded inverters are pre-existing / anomalous-since-start candidates, NOT degradation-over-time stories. Current baseline_excluded: INV 01.04.026, 01.05.030, 01.07.047, 01.07.051, 01.08.052, 01.08.059.
- Use latest_factor / latest_relative_factor for health/degradation explanations.

## Non-fabrication policy (Rafay's rule)

- EUR: we do NOT have the real feed-in tariff file imported. EUR is `lost_kwh * TARIFF_EUR_PER_KWH` with a clearly-labeled ASSUMED tariff (config `TARIFF_EUR_PER_KWH`, default 0.10). Every EUR value is tagged as assumption. Replace once Max adds real `lost_eur`.
- Repair-vs-replace: NO cost data exists -> qualitative recommendation only (driven by status, factor, event mix, baseline_excluded), explicitly labeled "no cost data".
- Error-code descriptions / tickets: `errorcodes description.xlsx` and `Tickets.xlsx` not imported locally -> agents note when context is unavailable rather than guessing.

## Stack decisions

- LLM: Gemini via `google-genai` SDK (key in `.env` as `GEMINI_API_KEY`). Model default `gemini-2.5-flash` (config `GEMINI_MODEL`).
- Everything degrades gracefully WITHOUT a key: deterministic templates produce findings, timelines, emails; only the polished prose / chatbot need the LLM.
- No pandas/duckdb dependency in the agent layer; reads JSON with stdlib, queries in-memory Python. Keeps it robust.
- API: FastAPI (`backend/agents/api.py`), default port 8088.
- Multi-plant: adapter supports Plant A (`frontend/public/data/`) and Plant B (`frontend/public/data_plant_b/`). All API endpoints accept `?plant=A|B`.

## API endpoints

```
GET  /                          → SolarTwin.html (test 2/project/ served as static)
GET  /api/health                → {status, llm, model}
GET  /api/plant?plant=A|B       → {summary, metadata, context, tariff}
GET  /api/map?plant=A|B         → inverter spatial map tiles
GET  /api/contacts              → all Enerparc contacts
GET  /api/findings?plant=A|B    → all findings (cached per plant, ?refresh=true to bust)
GET  /api/findings/{finding_id}?plant=A|B
GET  /api/timeline/{inverter_id}?plant=A|B&llm=true
POST /api/dispatch              → {finding_id, use_llm}
POST /api/chat                  → {message, history}
```

## Run commands

```bash
source .venv/bin/activate

# (optional) regenerate Max's payloads if the model run changed:
python scripts/export_frontend_payloads.py

# build findings + timelines + email drafts headlessly:
python scripts/run_agents.py                 # full run, LLM polish on top findings if key set
python scripts/run_agents.py --no-llm        # deterministic only
python scripts/run_agents.py --top 5         # control how many to dispatch/timeline

# serve API + frontend:
./start.sh                                   # → http://localhost:8088
# or manually:
uvicorn backend.agents.api:app --host 0.0.0.0 --port 8088 --reload

# (legacy) old Vite console:
cd frontend/agent-console && npm install && npm run dev
```
